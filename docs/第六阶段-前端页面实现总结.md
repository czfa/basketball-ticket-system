## 三、技术特点

### 3.1 技术栈

- **Vue 3**: 使用 Composition API，提高代码可维护性
- **Vue Router 4**: 单页面应用路由管理，支持路由守卫
- **Axios**: HTTP 请求库，统一配置请求和响应拦截器
- **原生 CSS**: 不使用第三方 UI 框架，代码简洁，符合毕业设计要求

### 3.2 架构特点

- **前后端分离**: 通过 RESTful API 与后端通信
- **状态管理**: 使用 localStorage 存储用户状态（可扩展为 Vuex）
- **会话管理**: 使用 Session Cookie 进行会话管理
- **路由守卫**: 实现用户和管理员权限控制
- **响应式设计**: 支持不同屏幕尺寸的设备访问

### 3.3 代码规范

- 使用 Vue 3 Composition API
- 文件命名使用 PascalCase（页面组件）
- 使用 Scoped CSS 避免样式污染
- 添加中文注释说明核心逻辑
- 代码结构清晰，便于维护

## 五、核心功能说明（论文可用）

### 5.1 选座购票流程（核心功能）

本系统的核心功能是在线选座购票，具体流程如下：

1. **赛事浏览**: 用户在赛事列表页浏览所有可购买的篮球赛事，可以通过搜索和状态筛选查找感兴趣的赛事。

2. **赛事详情查看**: 用户点击赛事卡片，查看赛事的详细信息（对阵双方、比赛时间、场馆、票价、可用座位数）。

3. **进入选座页面**: 用户点击"立即选座购票"按钮，系统检查用户登录状态，未登录则跳转到登录页，已登录则进入选座页面。

4. **座位选择**: 
   - 系统加载该赛事的座位列表，按区域（Zone）展示
   - 座位状态用不同颜色区分：绿色（可选）、蓝色（已选）、灰色（已售）、白色（不可用）
   - 用户可以通过区域筛选选择特定区域的座位
   - 用户点击可选座位，座位加入已选列表；再次点击已选座位，从列表中移除
   - 已售和不可用座位不可点击
   - 已选座位列表显示在右侧，包括座位信息和价格

5. **订单确认**: 用户选择完座位后，点击"确认选座并下单"，系统再次检查座位可用性（防止并发抢座）。如果所有座位可用，跳转到订单确认页；如果有座位不可用，提示用户并刷新座位状态。

6. **创建订单**: 在订单确认页，用户确认赛事信息、已选座位、订单金额，选择支付方式，点击"确认下单"创建订单。

7. **订单管理**: 订单创建成功后，用户可以在"我的订单"页查看所有订单，待支付订单可以取消或立即支付，已支付订单显示订单详情。

### 5.2 技术实现特点

- **并发控制**: 在下单前再次检查座位可用性，防止并发抢座导致数据不一致
- **状态管理**: 选座信息存储在 sessionStorage 中，页面刷新后仍然可用
- **用户体验**: 实时显示座位状态，已选座位高亮显示，操作反馈及时
- **数据一致性**: 座位状态与后端数据库保持一致，下单前再次验证

### 5.3 管理员功能

管理员端实现了数据概览功能，展示系统的统计数据（总赛事数、总订单数、总销售额、已售票数），为管理员提供系统运行情况的整体视图。

待实现的赛事管理、座位管理、订单管理功能将为管理员提供完整的后台管理能力，支持系统的日常运营和管理。

## 六、问题与解决方案

### 6.1 Element Plus 依赖问题

**问题**: 部分页面使用了 Element Plus 组件，但项目中没有安装。

**解决方案**: 
1. 安装 Element Plus（推荐，提高开发效率）
2. 改为原生 HTML/CSS（符合毕业设计要求，代码更简洁）

### 6.2 会话管理

**问题**: 前后端分离架构下，如何维护用户登录状态。

**解决方案**: 
- 使用 Session Cookie 进行会话管理
- 前端在 localStorage 中存储用户信息（用于路由守卫）
- 后端设置 Session Cookie，前端请求时自动携带

### 6.3 并发控制

**问题**: 多个用户同时选择同一座位，可能导致数据不一致。

**解决方案**: 
- 前端：在下单前再次检查座位可用性
- 后端：使用事务控制，确保订单创建和座位锁定的原子性
