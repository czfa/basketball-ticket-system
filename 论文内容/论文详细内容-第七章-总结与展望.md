### 7.1.2 解决的技术难点

#### 7.1.2.1 事务控制

**问题**：购票流程涉及多个数据库操作（创建订单、锁定座位、生成电子票），需要保证原子性。

**解决方案**：
- 使用Spring的`@Transactional`注解实现事务控制
- 确保订单创建、座位锁定、电子票生成在同一事务中完成
- 如果任何一步失败，整个事务回滚，保证数据一致性

**实现代码**：
```java
@Transactional(rollbackFor = Exception.class)
public Map<String, Object> createOrderWithSeats(Long userId, Long matchId, List<Long> seatIds) {
    // 创建订单
    // 锁定座位
    // 如果任何一步失败，事务回滚
}
```

#### 7.1.2.2 防重复购票

**问题**：多个用户同时选择同一座位，可能导致重复购票。

**解决方案**：
- **业务层检查**：下单前检查座位可用性
- **数据库层锁定**：使用`UPDATE ... WHERE is_booked = 0`实现悲观锁
- **事务保护**：订单创建和座位锁定在同一事务中完成

**实现代码**：
```java
// 业务层检查
Map<String, Object> availabilityCheck = seatService.checkSeatsAvailability(seatIds);

// 数据库层锁定（悲观锁）
QueryWrapper<Seat> wrapper = new QueryWrapper<>();
wrapper.eq("seat_id", seatId)
       .eq("is_booked", 0)
       .eq("is_available", 1);
int updated = seatMapper.update(seat, wrapper);
```

#### 7.1.2.3 密码加密

**问题**：用户密码需要安全存储，不能明文保存。

**解决方案**：
- 使用Spring Security的BCryptPasswordEncoder进行密码加密
- BCrypt是单向哈希算法，即使数据库泄露，也无法还原明文密码

**实现代码**：
```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

// 注册时加密密码
String encodedPassword = passwordEncoder.encode(user.getPassword());
user.setPassword(encodedPassword);
```

#### 7.1.2.4 前后端分离架构

**问题**：前后端分离架构下，如何实现用户认证和权限控制。

**解决方案**：
- 使用Session进行会话管理
- 前端在localStorage中存储用户信息，用于路由守卫
- 后端设置Session Cookie，前端请求时自动携带
- 使用CORS配置解决跨域问题

## 7.2 系统特点与创新点

### 7.2.1 系统特点

#### 7.2.1.1 在线选座功能

系统提供了直观的在线选座功能，用户可以：
- 在座位图上直观地查看座位分布
- 实时查看座位状态（可选/已选/已售/不可用）
- 按区域筛选座位
- 同时选择多个座位
- 查看已选座位列表和总金额

**技术实现**：
- 使用Vue 3的响应式数据，实时更新座位状态
- 使用CSS实现座位图的视觉效果
- 使用sessionStorage保存选座信息

#### 7.2.1.2 防重复购票机制

系统实现了业务层和数据库层双重防重复购票机制：
- **业务层**：下单前检查座位可用性
- **数据库层**：使用悲观锁，确保座位状态更新的原子性
- **事务保护**：订单创建和座位锁定在同一事务中完成

**技术优势**：
- 有效防止并发抢座导致的数据不一致
- 保证高并发场景下的数据正确性

#### 7.2.1.3 事务控制

系统在关键业务操作中使用事务控制：
- 购票流程：订单创建 + 座位锁定
- 支付流程：订单状态更新 + 电子票生成 + 赛事座位数更新
- 取消订单：订单状态更新 + 座位释放

**技术优势**：
- 保证数据一致性
- 避免部分操作成功、部分操作失败的情况

#### 7.2.1.4 前后端分离架构

系统采用前后端分离架构：
- 前端使用Vue 3构建，提供良好的用户体验
- 后端使用Spring Boot提供RESTful API
- 前后端通过HTTP/JSON进行通信

**技术优势**：
- 前后端可以并行开发，提高开发效率
- 前后端代码分离，易于维护和扩展
- 技术选型灵活，可以选择最适合的技术栈

### 7.2.2 创新点

1. **双重防重复购票机制**：结合业务层检查和数据库层锁定，有效防止并发问题。

2. **事务控制的完整应用**：在购票、支付、取消订单等关键流程中，全面使用事务控制，保证数据一致性。

3. **直观的在线选座界面**：使用颜色区分座位状态，提供良好的用户体验。

4. **完整的测试覆盖**：进行了功能测试、性能测试和安全性测试，确保系统质量。

## 7.3 不足与改进方向

### 7.3.1 发现的不足

#### 7.3.1.1 订单号生成策略

**问题**：当前使用时间戳+随机数生成订单号，在高并发场景下可能产生重复订单号。

**改进方向**：
- 使用雪花算法（Snowflake ID）生成唯一ID
- 或使用数据库序列生成订单号
- 在数据库层面添加唯一索引，防止重复

#### 7.3.1.2 分页功能

**问题**：当前手动实现分页，效率较低。

**改进方向**：
- 使用MyBatis-Plus的分页插件
- 配置Page对象，自动处理分页逻辑
- 支持更灵活的分页参数

#### 7.3.1.3 订单过期处理

**问题**：订单过期后，座位仍然被锁定，需要手动处理。

**改进方向**：
- 实现定时任务，定期检查过期订单
- 自动取消过期订单，释放座位
- 可以使用Spring的@Scheduled注解实现

#### 7.3.1.4 权限控制

**问题**：当前在Controller中手动检查Session，代码重复。

**改进方向**：
- 使用拦截器统一检查管理员权限
- 或使用Spring Security的权限注解（@PreAuthorize）
- 提高代码的可维护性

#### 7.3.1.5 电子票二维码

**问题**：当前未实现电子票二维码生成功能。

**改进方向**：
- 使用二维码生成库（如ZXing）生成二维码
- 二维码包含票号、座位信息等
- 支持扫码检票功能
