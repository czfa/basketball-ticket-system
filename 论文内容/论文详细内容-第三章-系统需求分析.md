# 第三章 系统需求分析

## 2 系统需求分析

### 2.1 问题描述

随着互联网技术的快速发展和体育产业的蓬勃兴起，传统的票务销售方式已经无法满足现代消费者的需求。传统的票务销售主要存在以下问题：

#### 2.1.1 购票效率低下

传统购票方式需要消费者前往售票点排队购票，耗费大量时间和精力。特别是在热门赛事开售时，往往出现排长队、一票难求的情况。消费者需要提前到达售票点，等待数小时甚至更长时间，严重影响了购票体验。

#### 2.1.2 座位选择不直观

传统购票方式中，消费者无法直观地看到场馆座位分布情况，只能通过座位号进行选择，难以判断座位的实际位置和视野效果。消费者往往需要依赖场馆平面图或咨询售票人员，信息获取不便，容易选择到不理想的座位。

#### 2.1.3 信息不透明

消费者难以实时了解座位的可用情况、价格信息等，容易错过心仪的座位。传统方式下，座位信息更新不及时，消费者无法及时了解哪些座位已被售出，哪些座位仍然可用，导致购票决策困难。

#### 2.1.4 管理成本高

对于票务管理方而言，传统方式需要大量人力进行售票、检票等工作，管理成本高，且容易出现人为错误。人工售票效率低，容易出现售票错误、座位重复销售等问题，影响票务管理的准确性和效率。

#### 2.1.5 数据统计困难

传统方式难以对销售数据进行实时统计和分析，不利于制定营销策略和优化运营。票务管理方无法及时了解销售情况、热门赛事、用户偏好等信息，难以进行数据驱动的决策。

#### 2.1.6 系统解决方案

为了解决上述问题，本系统采用Web技术实现票务信息的数字化管理，为消费者提供便捷的购票体验，同时也为管理方提供高效的管理工具。系统通过在线选座、实时库存更新、自动化处理等功能，有效解决了传统票务销售方式的痛点。

### 2.2 系统业务描述

#### 2.2.1 业务场景

本系统是一个篮球票务销售系统，主要服务于篮球赛事的票务销售和管理。系统涉及两类主要用户：

1. **普通用户**：需要购买篮球赛事门票的消费者
2. **管理员**：负责赛事管理、座位管理、订单统计的票务管理人员

#### 2.2.2 核心业务流程

**用户购票流程**：

1. **浏览赛事**：用户访问系统，浏览可购买的篮球赛事列表，可以按状态筛选、搜索赛事，查看赛事详情。

2. **选择座位**：用户选择感兴趣的赛事，进入选座页面。系统展示该赛事的座位分布图，用户可以直观地查看座位位置和状态，选择心仪的座位。

3. **创建订单**：用户选择座位后，确认选座信息，系统创建订单并锁定所选座位。订单创建后，用户需要在30分钟内完成支付。

4. **支付订单**：用户选择支付方式，完成订单支付。支付成功后，系统生成电子票，更新赛事可用座位数。

5. **查看订单**：用户可以随时查看自己的订单列表和订单详情，了解订单状态、座位信息等。

**管理员管理流程**：

1. **赛事管理**：管理员可以添加、编辑、删除赛事，更新赛事状态，设置赛事信息（对阵双方、比赛时间、场馆、票价等）。

2. **座位管理**：管理员可以为赛事批量生成座位，配置座位区域、类型、价格等信息。可以编辑单个座位信息，删除座位。

3. **订单统计**：管理员可以查看所有订单，进行销售统计，包括总销售额、订单数量、票务统计等。可以查看某场赛事的详细销售数据。

#### 2.2.3 业务特点

1. **在线选座**：提供直观的座位图，用户可以实时查看座位状态，选择心仪座位。

2. **实时库存**：座位状态实时更新，用户可以看到最新的座位可用情况。

3. **防重复购票**：通过业务层和数据库层双重保护，防止多个用户同时购买同一座位。

4. **事务保护**：关键业务操作（创建订单、支付、取消订单）都在事务中完成，保证数据一致性。

5. **自动化处理**：系统自动处理订单创建、座位锁定、电子票生成等操作，减少人工干预。

### 2.3 功能需求分析

#### 2.3.1 角色分析

系统涉及两类主要角色：普通用户和管理员。不同角色具有不同的权限和功能。

##### 2.3.1.1 普通用户角色

**角色定义**：需要购买篮球赛事门票的消费者。

**角色特征**：
- 可以注册账号，成为系统用户
- 可以浏览赛事信息，无需登录
- 需要登录后才能进行购票操作
- 只能查看和操作自己的订单

**主要权限**：
1. **用户认证**：
   - 用户注册：输入用户名、密码、邮箱、手机号等信息进行注册
   - 用户登录：使用用户名和密码登录系统
   - 用户登出：主动登出系统
   - 个人信息管理：查看和修改个人信息、修改密码

2. **赛事浏览**：
   - 浏览赛事列表：查看所有可购买的篮球赛事，支持分页和筛选
   - 查看赛事详情：查看赛事的详细信息（对阵、时间、场馆、票价、座位情况等）
   - 搜索赛事：根据关键词搜索赛事
   - 查看热门赛事：查看热门赛事推荐

3. **在线选座**：
   - 查看座位列表：查看某场赛事的所有座位
   - 查询可用座位：查询可售座位，支持按区域、类型、价格筛选
   - 选择座位：在座位图上直观地选择座位，可以同时选择多个座位
   - 查看已选座位：查看已选座位列表和总金额

4. **购票下单**：
   - 创建订单：选择座位后创建订单，系统锁定座位
   - 支付订单：对待支付订单进行支付，支付成功后生成电子票
   - 取消订单：取消待支付订单，释放已锁定座位

5. **订单查询**：
   - 查看订单列表：查看自己的所有订单，支持状态筛选
   - 查看订单详情：查看订单的详细信息（订单号、状态、座位列表、金额等）

##### 2.3.1.2 管理员角色

**角色定义**：负责赛事管理、座位管理、订单统计的票务管理人员。

**角色特征**：
- 拥有管理员账号，具有管理员权限
- 可以管理所有赛事和座位
- 可以查看所有用户的订单
- 可以进行销售统计和数据分析

**主要权限**：
1. **管理员认证**：
   - 管理员登录：使用管理员账号和密码登录系统
   - 管理员登出：主动登出系统
   - 查看个人信息：查看管理员个人信息

2. **赛事管理**：
   - 查看所有赛事：查看系统中的所有赛事
   - 新增赛事：添加新的赛事，设置赛事信息
   - 更新赛事：修改赛事信息
   - 删除赛事：删除赛事（需检查是否有订单）
   - 更新赛事状态：更新赛事状态（即将开始/进行中/已结束/已取消）

3. **座位管理**：
   - 批量生成座位：为赛事批量生成座位，配置区域、类型、价格等
   - 查看座位列表：查看某场赛事的所有座位
   - 更新座位信息：修改单个座位的信息（价格、类型、可用状态等）
   - 删除座位：删除某场赛事的所有座位

4. **订单管理与统计**：
   - 查看所有订单：查看系统中所有用户的订单，支持状态筛选
   - 销售统计：查看总销售额、订单数量统计、票务统计等
   - 赛事销售统计：查看某场赛事的详细销售数据（销售额、票数、订单数、上座率等）
   - 基础数据统计：查看赛事统计、订单统计、票务统计等

##### 2.3.2.2 用户用例详细描述

**用例1：用户注册**

- **用例名称**：用户注册
- **参与者**：用户
- **前置条件**：用户访问注册页面
- **主流程**：
  1. 用户输入用户名、密码、邮箱、手机号等信息
  2. 系统验证用户名和邮箱的唯一性
  3. 系统对密码进行BCrypt加密
  4. 系统创建用户记录，状态默认为"ACTIVE"
  5. 系统返回注册成功信息
- **异常流程**：
  - 用户名已存在：提示"用户名已存在"
  - 邮箱已存在：提示"邮箱已被注册"
  - 密码不符合要求：提示"密码不符合要求"
- **后置条件**：用户账号创建成功，可以登录系统

**用例2：用户登录**

- **用例名称**：用户登录
- **参与者**：用户
- **前置条件**：用户已注册
- **主流程**：
  1. 用户输入用户名和密码
  2. 系统验证用户名和密码（BCrypt验证）
  3. 系统创建Session会话
  4. 系统更新最后登录时间
  5. 系统返回登录成功，跳转到首页
- **异常流程**：
  - 用户名不存在：提示"用户名或密码错误"
  - 密码错误：提示"用户名或密码错误"
  - 账号被禁用：提示"账号已被禁用"
- **后置条件**：用户登录成功，可以访问需要登录的功能

**用例3：浏览赛事**

- **用例名称**：浏览赛事
- **参与者**：用户
- **前置条件**：无（可以未登录）
- **主流程**：
  1. 用户访问赛事列表页面
  2. 系统查询赛事列表（支持分页和状态筛选）
  3. 系统返回赛事列表，按时间排序
  4. 用户点击赛事，查看详情
  5. 系统返回赛事详细信息（对阵、时间、场馆、票价、座位情况等）
- **后置条件**：用户了解赛事信息，可以决定是否购票

**用例4：选座购票**

- **用例名称**：选座购票
- **参与者**：用户
- **前置条件**：用户已登录
- **主流程**：
  1. 用户选择赛事，进入选座页面
  2. 系统加载该赛事的座位列表，展示座位图
  3. 用户在座位图上选择座位，系统将座位加入已选列表
  4. 用户确认选座，点击"下单"
  5. 系统再次检查座位可用性（防止并发抢座）
  6. 系统创建订单并锁定座位（事务控制）
  7. 系统跳转到订单确认页面
  8. 用户选择支付方式，确认支付
  9. 系统处理支付，生成电子票（事务控制）
  10. 系统更新赛事可用座位数
  11. 系统返回支付成功信息
- **异常流程**：
  - 座位已被其他用户选择：提示"座位已被选择，请重新选择"
  - 订单创建失败：提示"订单创建失败，请重试"
  - 支付失败：提示"支付失败，请重试"
  - 订单过期：提示"订单已过期，请重新下单"
- **后置条件**：订单创建成功，座位被锁定；支付成功后，电子票生成

**用例5：查看订单**

- **用例名称**：查看订单
- **参与者**：用户
- **前置条件**：用户已登录
- **主流程**：
  1. 用户访问"我的订单"页面
  2. 系统查询该用户的所有订单（支持状态筛选）
  3. 系统返回订单列表，按创建时间倒序排列
  4. 用户点击订单，查看详情
  5. 系统返回订单详细信息（订单号、状态、座位列表、金额、支付信息等）
  6. 待支付订单可以取消或立即支付
  7. 已支付订单可以查看电子票信息
- **后置条件**：用户了解订单状态，可以进行相应操作

### 2.4 系统非功能性需求

#### 2.4.1 性能需求

1. **响应时间要求**：
   - 页面加载时间：< 2秒
   - API响应时间：< 500ms（正常情况）
   - 数据库查询时间：< 100ms（单表查询）
   - 选座页面座位数据加载时间：< 1秒

2. **并发处理能力**：
   - 系统需要支持至少100个并发用户同时访问
   - 购票操作需要支持至少10个并发请求同时处理
   - 需要实现防重复购票机制，避免并发问题导致的数据不一致
   - 系统在高并发场景下应保持稳定，不出现崩溃或数据错误

3. **数据处理能力**：
   - 支持单场赛事至少1000个座位
   - 支持至少10000个用户注册
   - 支持至少10000个订单
   - 支持至少100场赛事
   - 数据库查询应支持分页，避免一次性加载大量数据

#### 2.4.2 安全性需求

1. **用户认证安全**：
   - 密码必须加密存储，使用BCrypt算法，加密强度为10
   - 使用Session进行会话管理，Session ID随机生成
   - 登录状态需要定期验证，Session过期时间设置为30分钟
   - 防止暴力破解：连续登录失败5次后，账号临时锁定10分钟

2. **数据安全**：
   - 敏感信息（如密码）不能明文存储，必须加密
   - 使用HTTPS协议传输数据（生产环境）
   - 防止SQL注入攻击：使用参数化查询，不使用字符串拼接
   - 防止XSS攻击：对用户输入进行转义处理
   - 数据库连接使用连接池，防止连接泄露

3. **业务安全**：
   - 防止重复购票：业务层检查座位可用性，数据库层使用悲观锁
   - 订单创建和座位锁定需要在事务中完成，保证原子性
   - 用户只能查看和操作自己的订单，不能访问其他用户的订单
   - 订单号唯一性：使用唯一索引保证订单号不重复
   - 订单过期处理：订单创建后30分钟未支付自动过期

4. **权限控制**：
   - 管理员功能需要管理员权限，普通用户无法访问
   - 用户功能需要用户登录，未登录用户不能访问
   - 使用路由守卫和接口权限验证，双重保护
   - 管理员操作记录日志，便于审计
